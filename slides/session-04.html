🔧 DBT Built-in Tests

4 Generic Tests cơ bản

🎯 unique

Kiểm tra column có giá trị duy nhất

tests: - unique

Use case: Primary keys, order_id, customer_id

# models/staging/schema.yml
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - unique

📌 not_null

Kiểm tra column không có NULL

tests: - not_null

Use case: Required fields, dates, amounts

columns:
  - name: order_id
    tests:
      - not_null
  - name: order_date
    tests:
      - not_null

🔗 relationships

Kiểm tra foreign key validity

tests: - relationships

Use case: Foreign key references

columns:
  - name: customer_id
    tests:
      - relationships:
          to: ref('stg_customers')
          field: customer_id

✅ accepted_values

Kiểm tra values trong tập cho phép

tests: - accepted_values

Use case: Status, type, category columns

columns:
  - name: order_status
    tests:
      - accepted_values:
          values: ['PENDING', 'PAID', 'SHIPPED', 'DELIVERED', 'CANCELLED']
          quote_values: true

🎯 Best Practices

  • Add unique và not_null cho TẤT CẢ primary keys
  • Test foreign keys với relationships
  • Define accepted_values cho status/type columns
  • Click vào cards để xem examples chi tiết

🛠️ Custom SQL Tests

Khi nào cần custom tests?

Generic tests không cover được complex business logic. Ví dụ:

  • Revenue không được âm
  • Order date không được ở tương lai
  • Discount không được vượt quá total
  • Custom KPI thresholds

Example: Positive Revenue Test

# tests/assert_positive_revenue.sql
-- Returns failing rows (should return 0 rows)

select
    order_date,
    total_revenue
from {{ ref('int_revenue__daily') }}
where total_revenue < 0

⚠️ Quy tắc quan trọng

Custom test nên TRẢ VỀ ROWS LỖI. Nếu test pass → không có rows nào returned.

More Custom Test Examples

Test: Order Date Not In Future

# tests/assert_order_date_not_future.sql

select
    order_id,
    order_at
from {{ ref('int_orders__deduped') }}
where order_at > now()

Test: Discount Not Exceeding Total

# tests/assert_discount_valid.sql

select
    order_id,
    gross_amount,
    discount
from {{ ref('int_orders__deduped') }}
where discount > gross_amount

Test: Daily Order Count Threshold

# tests/assert_min_daily_orders.sql

select
    order_date,
    order_count
from {{ ref('int_revenue__daily') }}
where order_count < 0
  and order_date >= date_trunc('month', current_date)

⚙️ Severity Levels

Warn vs Error

Severity Behavior Use Case Example
error Fails the dbt run Critical data quality issues Primary key violations
warn Shows warning, continues Non-critical issues Missing optional data

Configuring Severity

Inline Configuration

# models/staging/schema.yml
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - unique:
              severity: error
          - not_null:
              severity: error
      - name: email
        tests:
          - not_null:
              severity: warn  # Optional field

Global Configuration

# dbt_project.yml
tests:
  dbt_shopee_tiktok:
    +severity: warn  # Default severity

    # Override for specific tests
    critical:
      +severity: error

# Or by model
models:
  staging:
    +tests:
      +severity: error

🎯 Recommended Strategy

  • error: unique, not_null cho primary keys
  • error: relationships cho foreign keys
  • warn: accepted_values (data có thể thay đổi)
  • warn: optional fields
  • warn: business logic thresholds

    • unique + not_null cho order_id, product_id

  2. Add tests cho intermediate models:

Check your understanding: Khi test FAIL, điều gì xảy ra với pipeline?
A. dbt run luôn dừng lại, không chạy tiếp models
B. Với severity=error: dbt test fails; Với severity=warn: pipeline continues
C. dbt tự động fix data và chạy tiếp
D. Không có gì xảy ra, test chỉ để logging
